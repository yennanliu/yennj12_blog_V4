> 10,000 個內部員工用，一切都很順。  
> 百萬外部用戶第一天上線，系統在 30 分鐘內崩潰。  
> 「加更多機器」不是答案——  
> 正確的問題是：哪些地方讓你根本無法加機器？

---

## 面試情境

> **面試官：**「你幫一家金融公司做了內部員工 AI 助手，10,000 個內部用戶，系統很穩定。現在 CEO 決定把這個產品開放給外部客戶，目標是百萬 MAU（月活躍用戶）。你說需要重新設計架構。從哪裡開始？你會做哪些改動？」

---

## 一、為什麼 10K → 1M 不只是「加機器」

```
10K 內部用戶的隱性假設：

  用戶行為：
  ├── 行為模式可預測（9-18 點工作時間，流量曲線平滑）
  ├── 用量相對均勻（員工配額相似）
  └── 系統問題可以容忍（內部用戶願意等）

  系統設計：
  ├── Session State 可以存在記憶體（少數實例就夠）
  ├── 認證：單一 LDAP/SSO
  ├── 資料：只有公司內部資料，不需要租戶隔離
  └── SLA：P95 < 10s 內部用戶接受

1M 外部用戶帶來的新挑戰：

  ├── 流量尖峰不可預測（病毒式傳播 → 1 小時內 100x 流量）
  ├── 用戶行為多樣（某些用戶濫用，某些幾乎不用）
  ├── SLA 要求更嚴格（外部客戶不等待，直接離開）
  ├── 安全攻擊面大（外部包含惡意行為者）
  └── 成本可見性高（CFO 會問每個 MAU 的 AI 成本是多少）

三個關鍵結構性差距：

  差距 1：Stateful 設計（Session 存記憶體）→ Auto-Scale 失效
  差距 2：沒有速率限制 → 單個失控客戶端可以拖垮整個系統
  差距 3：沒有非同步架構 → 高延遲請求佔滿所有連接槽 → 503

這三個差距，加機器都解決不了。
```

## 二、三個擴展維度：流量、資料、成本

```
維度 1：流量擴展（Compute Scaling）

  10K 用戶的計算需求：
  假設 10% 同時在線 = 1,000 並發，每個請求平均 3 秒
  → 需要 ~50 個服務實例

  1M 用戶（直接乘 100）：
  100,000 並發 × 3 秒 → 需要 ~5,000 個實例

  問題：線性擴展在 AI 系統不成立

  ├── LLM API 有配額上限（加服務實例不等於 LLM 配額增加）
  ├── Vector DB 有 QPS 上限（索引大了搜尋也會變慢）
  └── 外部依賴（SAP、DB）有 Rate Limit

  正確思路——先分類流量：
  ├── 真正需要即時的請求（需要同步 LLM 呼叫）→ 優化，不能壓縮
  ├── 可以非同步的請求（長報告生成）→ 走 Queue
  └── 可以被快取的請求（重複語義查詢）→ 走快取

維度 2：資料擴展（Data Scaling）

  10K 用戶：Session 資料少，單一 DB 夠用

  1M 用戶的資料挑戰：
  ├── Session 資料：1M × 100KB = 100GB（需要分散式快取）
  ├── 對話歷史：DB 讀寫量 100x，需要讀寫分離
  └── 稽核日誌：每次 AI 互動都要記錄（合規要求）→ 大量寫入

維度 3：成本擴展（Cost Scaling）

  10K 用戶估算：10K × 50 queries/月 × $0.02 = $10,000/月

  1M 用戶（直接乘 100，沒有優化）：
  LLM 成本 = $1,000,000/月
  → 這個數字讓 CFO 立刻叫停

  成本控制三招（優先順序）：
  ├── 語意快取（重複語義查詢不打 LLM）→ 降低 40-60% 請求量
  ├── 模型路由（簡單問題用小模型）→ 降低單次成本 60-80%
  └── Token Budget（每月限制每用戶 Token 配額）→ 防止異常用戶
```

## 三、無狀態服務設計（Stateless Design）

```
問題：有狀態服務為什麼無法水平擴展

  有狀態服務（10K 內部系統常見）：

  User → 打到 Instance A（Session 存在 A 的記憶體）

  Auto-Scale 觸發，請求被分配到 Instance B：
  User → Instance B → 找不到 Session → 對話斷掉，用戶不滿

  這就是為什麼「加機器」反而讓用戶體驗更差。

無狀態架構設計：

  ┌──────────────────────────────────────────────────────────────┐
  │  Load Balancer（無狀態，任何請求可以打任何實例）                │
  └──────┬────────────────────────────────────────┬─────────────┘
         │                                        │
         ▼                                        ▼
  ┌──────────────┐                        ┌──────────────┐
  │  Agent       │                        │  Agent       │
  │  Instance A  │                        │  Instance B  │
  │（不存任何 State）                     │（不存任何 State）
  └──────┬───────┘                        └──────┬───────┘
         │                                        │
         └──────────────────┬─────────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │  外部 State 儲存         │
              │  ├── Session：Redis      │
              │  ├── 對話歷史：DB        │
              │  └── 用戶設定：DB        │
              └─────────────────────────┘

  實作原則：
  ├── 每個請求帶上 session_id（存在 JWT Token 或 Cookie）
  ├── 服務啟動時不載入任何用戶 State，全部從外部儲存讀取
  └── 服務重啟或被替換，用戶完全感知不到

  效益：
  ├── Auto-Scale 可以隨時增加或移除實例
  ├── Rolling Deploy 不中斷對話
  └── 多地部署（多 Region）可以共用同一個 State 儲存
```

## 四、非同步佇列架構（Queue-based Architecture）

```
問題：同步架構在高流量下的連接槽瓶頸

  同步架構（10K 用戶時夠用）：
  User Request → Agent → LLM Call（5s）→ Response

  1M 用戶，1% 同時在線 = 10,000 並發：
  └── 10,000 個請求都佔著 HTTP 連接等 LLM → 連接槽耗盡 → 503

非同步佇列架構：

  判斷標準：預估耗時 > 8 秒 → 非同步化

  適合非同步的 AI 任務類型：
  ├── 長報告生成（合約分析、財務摘要、會議記錄整理）
  ├── 批次文件處理（100 頁 PDF 的 Embedding 和索引）
  └── 非時效性分析（「幫我分析上個月的銷售趨勢」）

  架構設計：

  User Request（長任務）
         │
         ▼
  API Gateway
  ├── 判斷：即時任務？還是非同步任務？
  │   標準：預估耗時 > 8s → 非同步
         │
         ▼（非同步分支）
  Task Queue（Pub/Sub 或 SQS）
  └── 立刻回傳 task_id 給 User（HTTP 202 Accepted）
         │
         ▼
  Worker Pool（可獨立擴展，不影響 API Gateway）
  └── 從 Queue 取任務 → 執行 LLM 任務 → 寫結果到 DB
         │
         ▼
  通知機制（兩種選擇）：
  ├── 輪詢：User 用 task_id 每 5 秒查詢一次狀態
  └── WebSocket Push：任務完成後主動推送給 User

  效益對比：
  ┌────────────────┬──────────────────────┬──────────────────────┐
  │                │  同步架構             │  非同步佇列架構       │
  ├────────────────┼──────────────────────┼──────────────────────┤
  │  API Gateway   │  等 LLM，佔連接槽     │  立刻回 202，釋放連接 │
  │  LLM 配額耗盡  │  所有請求 503          │  任務排隊等待        │
  │  Worker 故障   │  請求全部失敗          │  任務重新入隊，重試   │
  │  用戶感知      │  盯著 Loading 轉圈    │  可以做其他事，等通知 │
  └────────────────┴──────────────────────┴──────────────────────┘
```

## 五、語意快取策略（Semantic Caching）

```
傳統快取的問題：
  Key-Value 快取：完全相同的 Query 才命中
  「今天天氣如何」和「今天的天氣是什麼」→ 兩個不同的 Key → 快取不命中
  → 但語義相同，應該回傳同一個結果

語意快取架構：

  User Query「今天天氣如何」
         │
         ▼
  Embedding Model（將 Query 向量化）
         │
         ▼
  向量快取 DB（搜尋語義相似的歷史 Query）
  ├── 找到相似度 > 0.95 的 Query → 直接回傳快取結果（< 100ms）
  └── 沒找到 → 打 LLM，將結果存入快取
         │
         ▼（未命中時）
  LLM API Call（正常流程）

快取命中率的業務影響：

  AI 系統中，重複語義的查詢佔比：
  ├── 客服 Bot（有限的常見問題）：60-80% 重複
  ├── 內部知識庫助手（員工問類似問題）：40-60% 重複
  └── 個人化 AI（每個用戶問不同問題）：10-20% 重複

多層快取策略：

  Layer 1：語意快取（Semantic Cache）
  ├── 命中率：30-60%（依場景）
  ├── 延遲：< 100ms（vs LLM 的 2-8s）
  └── 成本：省去 LLM API 費用

  Layer 2：應用層快取（Redis）
  ├── 對象：Tool 結果（SAP 庫存 5 分鐘 TTL、匯率查詢 1 小時 TTL）
  └── 命中率：60-70%（Tool 結果不需要每次重新取得）

  Layer 3：HTTP 層快取（CDN）
  └── 對象：不含個人化資訊的靜態 API 回應

  整體效益：
  無快取：100% 請求打 LLM
  有語意快取（命中率 50%）：50% 請求打 LLM，成本降低 50%
  有語意 + Tool 快取（組合命中率 65%）：35% 請求打 LLM，成本降低 65%

  快取的 Trade-off：
  ├── 優點：成本降低、延遲降低、LLM 配額壓力降低
  ├── 缺點：需要管理快取失效（即時性要求高的場景不適合）
  └── 快取毒化風險：如果快取了一個錯誤回答，需要有機制清除
```

## 六、三層速率限制（Rate Limiting）

```
為什麼外部系統必須有速率限制：

  失控的客戶端 Bug：
  └── 某個企業客戶的 App Bug，造成每個用戶每秒呼叫 100 次 API
      → 一個企業客戶的 Bug 拖垮整個平台的其他用戶

  惡意行為者：
  └── 外部系統有機器人爬蟲，嘗試用免費帳號耗盡系統資源

三層速率限制設計：

  層次              限制對象          限制值（示例）        目的
  ──────────────────────────────────────────────────────────────
  Global            整個系統          500,000 RPM           保護後端基礎設施
  ──────────────────────────────────────────────────────────────
  Tenant（企業客戶）每個企業客戶       10,000 RPM（依方案）  防止單一客戶影響其他人
  ──────────────────────────────────────────────────────────────
  User              每個終端用戶       100 RPM               防止惡意或失控的單一用戶
                                      10,000 Tokens/天      控制每用戶 LLM 成本

  超限行為：
  ├── HTTP 429 Too Many Requests
  ├── Retry-After header（告知何時可以重試）
  └── 不要直接 Block，讓客戶端知道重試時機

  分層計費的差異化限制：
  └── 付費方案（Enterprise）：配額 5x 免費方案
      付費客戶超限 → 警告並允許短暫超出（寬限），不直接 503
      免費用戶超限 → 嚴格限制，引導升級
```

## 七、資料庫擴展設計

```
10K 用戶可接受：單一 DB 讀寫混合

  問題：
  ├── 讀寫混合：讀取效能拖累寫入（鎖競爭）
  ├── 單點故障：DB 掛了，整個系統掛了
  └── 大表查詢：某些 Analytics Query 拖垮整個 DB

擴展策略：

  策略 1：讀寫分離（Read Replica）
  ├── 寫入：Primary DB（只有 Agent 的寫入走這裡）
  └── 讀取：Read Replica（1-3 個，依讀取壓力調整）

  AI 系統的讀寫比：讀取 >> 寫入
  └── Agent 讀取歷史對話（讀）>> 寫入新對話（寫）
      Read Replica 可以吸收 80% 的 DB 壓力

  策略 2：資料分層儲存

  資料類型              儲存選擇              理由
  ──────────────────────────────────────────────────────────────
  Session（熱資料）      Redis                低延遲（< 5ms），TTL 自動清理
  對話歷史（溫資料）     PostgreSQL           結構化，可查詢，讀寫分離
  向量索引               Vector DB            語意搜尋，不適合關聯式 DB
  稽核日誌（冷資料）     Object Storage       便宜（$0.02/GB），大量寫入不影響主 DB
  ──────────────────────────────────────────────────────────────

  策略 3：Connection Pool
  └── 1M 用戶 → 100 個服務實例，每個實例如果開 50 個 DB 連接
      = 5,000 個 DB 連接（超過 PostgreSQL 的最大連接數）
      → 使用 PgBouncer 做 Connection Pooling，集中管理連接數
```

## 八、系統效應：擴展設計的量化影響

```
維度              有擴展設計                     沒有擴展設計
──────────────────────────────────────────────────────────────────
流量峰值           Auto-Scale 自動應對，          預置容量，峰值超過上限
                   系統不下線                     → 503，用戶流失
──────────────────────────────────────────────────────────────────
LLM 成本           快取命中率 50%+，              100% 打 LLM，
                   每 MAU ~$1.5/月                每 MAU ~$4/月（2.7 倍）
──────────────────────────────────────────────────────────────────
系統穩定性         無狀態，Scale-Out 有效，        Stateful，新實例找不到 Session，
                   任意實例可處理任意請求          Scale-Out 反而讓用戶體驗更差
──────────────────────────────────────────────────────────────────
惡意用戶影響       Rate Limiting 隔離，            一個失控客戶端可以
                   不影響其他用戶                 拖垮整個平台
──────────────────────────────────────────────────────────────────
長任務體驗         非同步佇列，立刻回傳 202，       同步等待，
                   用戶可以繼續操作               連接 timeout → 錯誤
──────────────────────────────────────────────────────────────────
DB 瓶頸            讀寫分離 + 資料分層，           單一 DB 讀寫混合，
                   讀取不拖累寫入                  高峰期相互干擾，P99 爆炸
──────────────────────────────────────────────────────────────────

設計成本 vs 收益：

  成本：
  ├── Redis 叢集：~$200-500/月
  ├── Read Replica：~$300/月
  ├── Queue 服務：按訊息量計費，通常 < $100/月
  └── 工程時間：2-3 週實作這套架構

  收益：
  ├── LLM 成本降低 50%（快取）
  ├── 系統穩定性：流量 10x 時不崩潰
  └── 每個 MAU 的邊際成本下降，業務可以更快增長
```

## 九、面試答題要點

> *「這道題的核心不是『加幾台機器』，而是找出哪些架構假設在 10K 時成立、在 1M 時失效。*
>
> *四個關鍵設計：*
>
> *第一，無狀態服務。Session State 必須移出記憶體，放到 Redis 或外部 DB。讓 Load Balancer 可以把請求打到任何實例，Auto-Scale 才真正有效——否則 Scale-Out 反而讓用戶對話斷掉。*
>
> *第二，非同步佇列。預估耗時 > 8 秒的任務走 Queue，API Gateway 立刻回傳 task_id（HTTP 202），不佔連接槽。Worker Pool 依 Queue 深度獨立擴展。LLM 配額耗盡時任務排隊等待，不是 503。*
>
> *第三，語意快取。AI 系統的查詢中，30-60% 語義上是重複的。用 Embedding 做語意相似度搜尋，命中率 > 0.95 直接回傳快取。整體快取命中率可以達到 50%+，LLM 成本降低 50%。*
>
> *第四，三層速率限制。Global / Tenant / User 三層，保護後端和 LLM 配額，防止惡意用戶或失控客戶端影響其他用戶。*
>
> *成本可見性：CFO 會問每個 MAU 的 AI 成本。有快取的情況下，從 $4/MAU 降到 $1.5/MAU——這個數字要和業務可接受的定價模型對齊。」*
