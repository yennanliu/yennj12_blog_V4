> 「AI 系統出問題了」不是問題描述，是症狀描述。  
> 在沒有找到根因之前，任何修復都是猜測。  
> 結構化的故障排查不是把所有可能都試一遍，  
> 而是用最少的資訊快速縮小到一個節點。

---

## 面試情境

> **面試官：**「你們的 AI 客服系統，今天早上 9 點開始，客戶回報：系統變慢了，有時候會給出奇怪的答案。你不在現場，只能遠端處理。請一步一步說明你的排查思路。」

---

## 一、AI 系統故障排查的特殊性

```
傳統系統 vs AI 系統的故障特性：

  傳統系統的故障：
  ├── 確定性：相同輸入，相同輸出（Error 可重現）
  ├── 有明確的 Error Code（HTTP 500、DB Connection Failed）
  └── 對錯有明確定義（回傳 null 是 Bug）

  AI 系統的故障：
  ├── 不確定性：LLM 輸出有隨機性，問題可能間歇性出現
  ├── 「慢」和「奇怪答案」是兩個可能獨立的問題
  │   （同時發生不代表是同一個根因）
  ├── 多個潛在根因層（模型、Retrieval、工具 API、基礎設施）
  └── 「奇怪的答案」沒有 Error Code——需要評估，不是看 Log 就有答案

三個常見的排查誤區：

  誤區 1：「LLM 模型一定有問題」
           → 不一定。延遲問題通常是基礎設施；品質問題可能是 Retrieval 失敗

  誤區 2：「昨天還好的，一定是昨晚改了什麼造成的」
           → 不一定。流量模式改變也可能觸發潛在問題（如 Cold Start）

  誤區 3：「重啟一下應該就好了」
           → 可能暫時緩解，但根因沒找到，問題一定再發

AI 系統的一個關鍵洞察：
  Retrieval 失敗（搜尋返回 0 結果）→ LLM 沒有 Context → LLM 靠自己猜
  → 症狀：「奇怪的答案」，但 Error Log 沒有任何 Error
  這種問題，只有 Trace 才能發現。Log 完全看不出來。
```

## 二、結構化診斷框架（五步驟）

```
步驟 1：釐清症狀（Clarify Symptoms）——2 分鐘

  必問的六個問題：
  ├── 「多慢？」→ P50？P95？還是 timeout？（數字很重要）
  ├── 「奇怪答案是哪種？」→ 幻覺？不相關？截斷？
  ├── 「什麼時候開始？」→ 精確到分鐘，對應到部署記錄
  ├── 「什麼事件前後？」→ 昨晚有部署嗎？有流量異常嗎？
  ├── 「影響範圍？」→ 所有用戶？特定類型查詢？特定地區？
  └── 「頻率？」→ 100% 的請求？還是間歇性 10%？

  目標：把「系統很慢，答案很奇怪」轉化為：
  「從 09:00 開始，P95 延遲從 3s 升到 12s，
   所有用戶都受影響，答案品質差（答非所問），
   昨晚 20:00 有更新 RAG 索引。」

步驟 2：確認可觀測性資料（Check Observability Data）——3 分鐘

  按順序看：
  ├── Metrics（先看 Dashboard）
  │   ├── 確認問題確實存在（有時候是用戶主觀感受）
  │   ├── 問題開始的精確時間點（和部署時間對比）
  │   └── 影響的指標：Error Rate？P95 Latency？還是 Throughput 下降？
  │
  ├── Trace（幾條代表性的問題請求）
  │   └── 打開瀑布圖，找哪個 Span 佔了最多時間
  │       → 3 分鐘內可以定位「問題在哪一層」
  │
  └── Logs（有針對性地找，不是 grep 全部 log）
      └── 在問題開始的時間點，有沒有新的 Error Type 出現？

步驟 3：建立假說（Form Hypotheses）——1 分鐘

  不是列出所有可能，而是根據觀測資料提出 2-3 個最可能的假說。

  示例：
  「P95 延遲 09:00 開始升高，Trace 顯示 vector_search Span 從 200ms 到 9,800ms」
  → 假說 A：Vector DB 效能問題（Cold Start？索引重建？）
  → 假說 B：昨晚的新索引有問題（格式不對，搜尋失敗，返回 0 結果）

步驟 4：測試假說（Test Hypotheses）——依假說複雜度 5-15 分鐘

  原則：先測最快能得到答案的假說，不是先測最可能的假說。

  ├── 快速測試（1 分鐘）：直接呼叫疑似問題的服務，看延遲
  ├── 比較測試：今天的 Trace vs 昨天的 Trace，找差異點
  └── 隔離測試：繞過疑似問題元件（用 Mock），看症狀是否消失

步驟 5：根因確認與修復（Fix and Verify）

  ├── 在 Staging 環境驗證修復方案
  ├── 生產修復後，確認指標回到正常水位（不只是「感覺好了」）
  └── 寫 Post-Mortem：根因、時間線、影響範圍、預防措施
      （不是要懲罰誰，而是防止下一次）
```

## 三、症狀分類矩陣

```
症狀              最可能根因（按概率排序）        第一步要看的資料
──────────────────────────────────────────────────────────────────
P95 延遲升高      1. Vector DB 變慢               Trace 瀑布圖
                  2. 外部 API 變慢（SAP、DB）      vector_search Span 時長
                  3. LLM input_tokens 增加         LLM input_tokens 趨勢圖
                  4. Cold Start 增加（Scale-Out 時）
──────────────────────────────────────────────────────────────────
Error Rate 升高   1. 外部 API 故障或 Rate Limit    Error Log 的 error_type 分佈
                  2. LLM API 故障                  HTTP Status Code 分佈
                  3. Token 超限（輸入太長）         LLM input_tokens 峰值
                  4. DB 連接數耗盡
──────────────────────────────────────────────────────────────────
品質退化          1. Retrieval 召回率下降           Shadow Eval 分數趨勢
（答案奇怪）       2. LLM 模型版本靜默更新          用戶 Thumbs Down 率
                  3. Embedding Model 不一致         Context Recall 指標
                  4. Knowledge Base 資料過期
──────────────────────────────────────────────────────────────────
間歇性錯誤        1. 外部依賴不穩定（特定時段）      錯誤的時間分佈（有沒有規律？）
                  2. Memory Leak → OOM 重啟         服務重啟記錄
                  3. Race Condition（Parallel Agent）Trace 的並行 Span 分析
──────────────────────────────────────────────────────────────────
對話記憶消失      Session State 配置問題            Session 儲存的讀寫成功率
                  （Scale-Down / Scale-Out）         服務重啟時間點
──────────────────────────────────────────────────────────────────
```

## 四、系統分層分析（找出問題在哪一層）

```
AI 系統的五個分析層次：

  問題一定在這五層中的某一層。
  診斷策略：從外層開始，逐層排除，找到第一個異常的層。

  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 5：用戶端 / 網路                                        │
  │  ├── 檢查：用戶的網路、瀏覽器、DNS                             │
  │  └── 快速測試：curl 直接呼叫 API Gateway，延遲多少？           │
  │      如果 curl 正常 → 問題在 Layer 5（用戶端）                  │
  └──────────────────────────────────────────────────────────────┘
                               │ 排除 Layer 5
                               ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 4：API Gateway / Load Balancer                         │
  │  ├── 檢查：Gateway 的 P95 Latency log                         │
  │  └── 快速測試：Gateway 記錄的延遲 vs Agent 收到請求的時間差    │
  │      如果 Gateway → Agent 的時間差很大 → 問題在 Layer 4        │
  └──────────────────────────────────────────────────────────────┘
                               │ 排除 Layer 4
                               ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 3：Agent 服務本身                                       │
  │  ├── 檢查：Agent 的 CPU、Memory 使用率、實例數量               │
  │  └── 快速測試：Trace 瀑布圖，看 Agent 內部哪個 Step 最慢       │
  │      是 LLM Call？還是 Tool Call？還是 Retrieval？             │
  └──────────────────────────────────────────────────────────────┘
                               │ 發現是某個 Tool Call 慢
                               ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 2：外部依賴（LLM API、Vector DB、工具 API）              │
  │  ├── 檢查：各依賴服務的 Status Page / Health Endpoint         │
  │  └── 快速測試：直接呼叫 Vector DB API，測試標準查詢的延遲      │
  └──────────────────────────────────────────────────────────────┘
                               │ 排除 Layer 2
                               ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 1：資料儲存（DB、Object Storage）                        │
  │  ├── 檢查：DB 查詢延遲、連接數、磁碟 I/O、Slow Query Log       │
  │  └── 快速測試：執行診斷查詢，確認回應時間是否正常               │
  └──────────────────────────────────────────────────────────────┘

  分層排除的效率：
  不用猜「是哪裡的問題」，每一步都有測試結果告訴你「這一層正常還是異常」。
  平均 5-10 分鐘可以定位到具體層次。
```

## 五、五種 AI 系統常見失效模式

```
失效模式 1：Vector DB Cold Start（冷啟動延遲）

  症狀：每隔一段閒置後，第一個請求極慢（10-50x），之後恢復正常
  信號：Trace 的 vector_search Span 有規律尖峰（不是持續高延遲）
  根因：Vector DB 實例閒置後進入低功耗狀態，第一個查詢重新載入索引
  
  修復：
  ├── 短期：設定 min-instances = 1（確保始終有暖機實例）
  └── 長期：設定 Warm-up 排程（每 10 分鐘發送一個 Ping Query）

─────────────────────────────────────────────────────────────────

失效模式 2：Context Window 累積膨脹

  症狀：新對話的請求正常，長對話的請求逐漸變慢
  信號：LLM Span 的 input_tokens 隨對話輪次單調遞增，沒有上限
  根因：Conversation History 沒有截斷策略，累積到幾萬 Token
  
  修復：
  ├── 立刻：設定 Token Budget，History 超過上限時從最舊輪次截斷
  └── 長期：超過 10 輪的對話，先壓縮成摘要（見 Part 38）

─────────────────────────────────────────────────────────────────

失效模式 3：外部 API 間歇性超時（Circuit Breaker 缺席）

  症狀：Error Rate 升高，但不是 100%；特定時段更嚴重
  信號：tool.call Span 有高比例的 latency > 5,000ms，且 success=false
        错误集中在特定時段（例如月結日、備份窗口）
  根因：外部 API（SAP、Oracle）在特定時段負載高，Timeout 頻繁
  
  修復：
  ├── Tool Timeout 設定（SAP=5s, Oracle=3s）
  ├── Retry with Exponential Backoff（不要立刻重試）
  └── Circuit Breaker：連續失敗 5 次 → 開路 30 秒，不再呼叫

─────────────────────────────────────────────────────────────────

失效模式 4：Embedding Model 版本不一致

  症狀：Retrieval 品質下降，但 Vector DB 查詢延遲正常，無任何 Error
  信號：Context Recall 指標從 0.85 降到 0.45，但 vector_search 成功
         LLM 輸出開始出現幻覺（因為找到的文件不相關）
  根因：系統更新時 Embedding Model 版本從 v1 換到 v2，
        但 Vector Index 還是用 v1 建的
        → Query 向量在 v2 空間，Index 向量在 v1 空間
        → 餘弦相似度計算完全無意義
  
  修復：
  ├── 立刻：把 Query Embedding Model 改回 v1（環境變數，30 秒）
  └── 長期：建立 Embedding Model 版本和 Index 版本的綁定機制
             換 Embedding Model 必須重建全部 Index

─────────────────────────────────────────────────────────────────

失效模式 5：LLM 模型版本靜默更新

  症狀：品質緩慢退化，沒有明顯 Error，指標 7 日移動平均緩慢下滑
  信號：Shadow Faithfulness 分數從 0.87 緩慢降到 0.79（2 週內）
        用戶 Thumbs Down 率從 3% 升到 7%
  根因：使用了 model alias（如 gemini-pro）而非具體版本號
        服務商靜默更新了模型行為，與現有 Prompt 不相容
  
  修復：
  ├── 立刻：改為具體版本號（gemini-2.0-flash-001），釘選已驗證版本
  └── 長期：新模型版本更新流程：Staging 測試 → Eval Pipeline → Canary
```

## 六、假說驅動的診斷流程示例

```
症狀：「今天早上 9 點開始，系統變慢了，有時候給奇怪答案」

步驟 1：釐清症狀（2 分鐘）

  「多慢？」→ 平均從 3s 到 12s，有時 30s timeout
  「奇怪答案是哪種？」→ 答非所問，好像查不到資料就瞎猜
  「什麼時候開始？」→ 09:00，昨晚 20:00 有更新 RAG 索引
  「影響全部用戶？」→ 是的，所有查詢都有
  
  初步判斷：「RAG 索引更新」和「09:00 開始出問題」有時間關聯。

步驟 2：查可觀測性資料（3 分鐘）

  Metrics：P95 延遲確實從 09:00 開始升高 ✓
  Trace：打開一條慢請求
         → vector_search Span = 9,800ms（正常：200ms）✓
         → LLM Span 正常（280ms）
         → vector_search 回傳 top_k=5，結果 count=0（！）
  
  關鍵發現：Vector DB 查詢返回 0 個結果
            LLM 沒有 Context → 靠自己瞎猜 → 「奇怪答案」

步驟 3：建立假說（1 分鐘）

  假說 A：Vector DB 索引損壞（昨晚重建失敗？）
  假說 B：Embedding Model 版本不一致（新索引 v2，查詢用 v1？）

步驟 4：測試假說（5 分鐘）

  測試 A（1 分鐘）：直接查 Vector DB，返回 0 結果 → 索引存在，但搜尋無效
  測試 B（2 分鐘）：查昨晚的部署記錄 → 昨晚同時更新了 Embedding Model v1→v2
                    但 Vector Index 是用 v1 建的，還沒重建
  
  結論：假說 B 成立。Embedding Model 不一致導致查詢和索引不在同一向量空間。

步驟 5：修復（30 秒 + 1 小時）

  立刻：把 Query Embedding 改回 v1（改環境變數 → 重啟 Agent）
        → 30 秒後，P95 延遲恢復正常，答案品質回來
  
  長期：重建 Vector Index（用 v2 對所有文件重新 Embed）
        建立版本綁定機制，未來避免同樣問題
```

## 七、系統效應：結構化診斷的影響

```
維度              有結構化診斷框架              沒有結構化診斷框架
──────────────────────────────────────────────────────────────────
找到根因時間       5-15 分鐘                     幾小時（甚至找錯方向）
──────────────────────────────────────────────────────────────────
修復有效性         根因確認後修復，              猜測修復，
                   問題不再復發                  問題可能再發
──────────────────────────────────────────────────────────────────
誤操作風險         先測試假說再修復，            在壓力下可能做出
                   不輕易改動生產配置            有害的緊急變更
──────────────────────────────────────────────────────────────────
對外溝通品質       「問題在 Vector DB，          「還在查，我們盡力」
                   預計 30 分鐘內修復」          （客戶不安心）
──────────────────────────────────────────────────────────────────
事後學習           Post-Mortem 記錄精確根因，    同樣問題可能再發，
                   可以針對性加監控或防護        無法針對性改進
──────────────────────────────────────────────────────────────────
MTTR 趨勢          每次事故後 MTTR 縮短          MTTR 無規律，依賴個人經驗
（Mean Time to Resolve）（知識被系統化積累）      （人走了知識就走了）
──────────────────────────────────────────────────────────────────
```

## 八、面試答題要點

> *「聽到『慢了』和『奇怪的答案』，我的第一反應是：這是兩個獨立問題，還是同一個根因？我不會同時追兩個方向。*
>
> *第一步：釐清症狀。我會問：延遲具體是多少？P95 還是 P50？奇怪答案是幻覺、不相關、還是截斷？什麼時候開始？昨晚有沒有部署？影響全部請求還是特定類型？把模糊描述轉換成可測量的指標。*
>
> *第二步：打開可觀測性資料。先看 Metrics 確認問題存在和時間點，再看幾條 Trace 的瀑布圖，找哪個 Span 佔了最多時間。這樣我可以在 5 分鐘內定位到問題在哪一層——是 Retrieval 慢、LLM 慢、還是外部 API 慢。*
>
> *AI 系統的特殊診斷邏輯：「奇怪的答案」的根因可能是 Retrieval 完全失敗（vector_search 返回 0 結果），而不是 LLM 本身的問題。LLM 在沒有 Context 的情況下靠自己猜，症狀是「答非所問」，但 Error Log 沒有任何錯誤。只有 Trace 才能發現這個問題。*
>
> *第三步：建立假說，先測試最快能得到答案的假說，而不是先測最可能的。確認根因後，先做最快的短期修復（環境變數切換），再計劃長期修復（索引重建），並用指標確認修復有效——不只是主觀感覺「好了」。」*
